# M4.0 spike — semantic-layer protocol note

Status: **complete** (2026-07-13). Two halves: (a) Claude Code hook
contract (verified against the official hooks docs), (b) daemon ingest +
pane identity (verified against the code). This note is the spike
deliverable per PLAN.md §M4.0. Verdict: **the M4 design holds** — the
blocking-approval model is directly supported, with a safe failure mode.
The one empirical smoke test (blocking hook, decision honored, safe
fallback) was run and **passed** on 2026-07-14 — see "Empirical check".

## (b) Daemon side — verified against the code

### Ingest socket

`src/admin.rs` is the template and needs no invention: line-JSON over a
Unix socket in the state dir, 0600, plus a peer-uid check (`peer_cred()`
== socket owner) as defence in depth. The ingest socket is a **separate
socket** (`ingest.sock`): admin accepts *commands* from the owner, ingest
accepts *data* from same-uid processes (shell hooks, agent hooks). Keeping
them apart means the ingest schema can stay strict and rate-limited
without touching the admin path, and a compromised/confused hook script
can never reach `revoke`/`pair`.

Differences from admin.sock, per the Codex review:
- strict serde schema (`deny_unknown_fields`), per-field length caps
- rate limit per connection + global pending-event cap
- requests carry no authority: an event can raise attention, never act

`remux emit` (new CLI subcommand) wraps "connect, write one JSON line,
read ack" so hook snippets stay one-liners — same shape as
`admin::request()`.

### Pane identity

- Shell/agent hooks know their pane as `$TMUX_PANE` (tmux sets it in every
  pane environment; format `%N`, stable for the pane's lifetime).
- **Gap found:** topology's `PaneInfo` (`src/tmux.rs:258`) carries
  `index/active/command` but not the pane id — the `list-panes` format at
  `src/tmux.rs:311` is `#{pane_index} #{pane_active}
  #{pane_current_command}`. M4a adds `#{pane_id}` to the format and struct
  (`%`-prefixed, no escaping concerns — it's numeric after `%`).
- Mapping an event to a session = look up `%N` in the current topology
  snapshot (`app.topology` watch channel already holds it). Pane died →
  id absent from snapshot → pending requests for it expire immediately
  (M4b registry hooks into the same dirty-bit republish).

### Byte-source confirmation (why out-of-band is required)

Re-verified the review's blocker directly: `src/ws.rs:193` opens the PTY
inside the per-connection websocket handler and it dies with the socket —
no client connected ⇒ no byte stream at all, and the stream that does
exist is tmux's rendered view of the *active window only*. Any in-band
marker parsing would be blind exactly when push notifications matter.
Attention today survives this because it polls `window_activity` via the
tmux CLI (`src/attention.rs`), i.e. it is already out-of-band.

## (a) Claude Code hook contract — verified

Source: official hooks docs (code.claude.com/docs/en/hooks), checked
2026-07-13.

### The two hooks map exactly onto M4a / M4b

**M4a (attention) → `Notification` hook.** Display-only by design ("exit
code 2 or JSON decision have no effect"), fires with
`notification_type: "permission_prompt"` when a permission dialog appears
and `"idle_prompt"` when Claude finishes and waits (also
`agent_needs_input`, `agent_completed`, and others). Payload includes
`session_id`, `cwd`, `message`. Perfect for `remux emit`: fire-and-forget,
no blocking, no decision authority — matches the "informational" class.

**M4b (approvals) → `PermissionRequest` hook.** A dedicated event that
fires when the permission dialog is shown (distinct from `PreToolUse`).
The hook process blocks; its stdout JSON decides:

```json
{"hookSpecificOutput": {"hookEventName": "PermissionRequest",
  "decision": {"behavior": "allow" | "deny"}}}
```

Binary — no "ask"/"defer" at this event. stdin context: `session_id`,
`prompt_id`, `transcript_path`, `cwd`, `permission_mode`, `tool_name`,
`tool_input`. Note `$TMUX_PANE` is *not* in the payload — the hook script
reads it from its own environment (hooks run in the session's context)
and sends both.

### Blocking budget and failure mode (the load-bearing facts)

- Default hook timeout: **600s**; per-hook override via `"timeout"`
  (seconds) in the hook config. 30–120s of blocking is squarely
  supported.
- **On timeout, the hook's output is ignored and the interactive
  permission dialog appears as if the hook had not run.** Same for
  non-zero exit. This is exactly the failure mode we want: remux
  unreachable / phone silent / card expired → Claude falls back to asking
  on the Mac. **No path auto-denies or auto-allows on failure.**
  Consequence: `remux emit --wait` exits non-zero (no stdout) when the
  daemon reports expiry — never fabricates a decision.
- Card expiry must be comfortably below the hook timeout (e.g. hook
  timeout 120s, card expiry ~100s) so the fallback is the hook exiting
  cleanly, not Claude killing it.

### Caveats found

- `PermissionRequest` hooks **do not fire in non-interactive mode**
  (`-p`). Fine for remux's use case (interactive tmux sessions); recorded
  so nobody "fixes" it later with PreToolUse without reading this: a
  PreToolUse deny is a hard block, changing the failure mode.
- Docs don't specify merging when *multiple* PermissionRequest hooks
  return conflicting decisions (PreToolUse is documented as
  most-restrictive-wins). remux installs one hook; if a user has others,
  behavior is theirs to own — document in the install snippet.
- Docs don't specify exit-0-with-no-JSON; we use non-zero exit for
  "no decision" anyway.

### Empirical check — DONE (2026-07-14, Claude Code v2.1.197)

Ran the deferred smoke test on a real interactive session (isolated tmux
`-L hooktest`, scratch project, a `PermissionRequest` matcher on `Bash`
whose command hook sleeps 90s then prints an `allow` decision). Every
load-bearing assumption held:

1. **Blocking works and is honored.** Hook fired, slept exactly 90s, the
   permission dialog waited the whole time, then `{"decision":{"behavior":
   "allow"}}` was accepted — the transcript shows `Ran 1 shell command —
   Allowed by PermissionRequest hook` and the file was created. No
   send-keys, pure decision JSON. The minimal payload (no `updatedInput`)
   is accepted.
2. **The interactive dialog is shown concurrently while the hook blocks**,
   not hidden behind it. Important for M4b UX: the Mac shows its own
   dialog the entire time the phone card is pending; whichever surface
   resolves first wins. Either can approve — good, but the card registry
   must tolerate the Mac having already answered (atomic consume, and a
   remote decision arriving for an already-resolved prompt is a no-op).
3. **Hook death ⇒ interactive fallback stays fully live.** Killed the
   blocking hook mid-sleep (proxy for timeout / crash / `emit --wait`
   exiting non-zero on expiry). The on-screen dialog remained interactive
   and neither auto-allowed nor auto-denied. Manually selecting **No**
   genuinely denied (file not created); selecting **Yes** genuinely
   allowed. Confirms the "never fabricate a decision on failure" property
   the whole safety story rests on.

Payload fields captured from a live fire (superset of what the docs list):
`session_id`, `transcript_path`, `cwd`, `prompt_id`, `permission_mode`,
`effort`, `hook_event_name`, `tool_name`, `tool_input`, and a
`permission_suggestions` array (addDirectories / setMode). `$TMUX_PANE`
is *not* in the payload (confirmed) — the hook reads it from its own env.

Gotcha for the driver, not the design: batched `Down Down Enter` sent as
one string to the fallback dialog misfired once (landed on Yes). Spaced,
verified-per-step keystrokes selected correctly. Irrelevant to remux
(real users press keys), noted so the next person testing send-keys
doesn't chase a phantom.

## Protocol sketch (input to M4a/M4b design)

```
shell/agent hook ──(one JSON line)──▶ ingest.sock ──▶ event bus
                                                      │ class: informational → attention/feed
remux emit --wait ◀──(decision line)── pending-card registry (M4b)
                                          ▲ one-shot id, expiry, atomic consume
PWA ◀── WS card frame / deep-link fetch ──┘   Approve/Deny (device needs `approve`)
```

Event line (ingest, strict schema, `deny_unknown_fields`):
`{"v":1, "kind":"agent_needs_input"|"agent_permission"|"command_started"|
"command_finished", "pane":"%3", "source":"claude-code"|"shell", ...}`
— `agent_permission` (M4b) is the only kind that opens a pending card and
holds the connection for the decision; everything else acks immediately.
